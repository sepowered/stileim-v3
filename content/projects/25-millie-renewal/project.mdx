## **이번 프로젝트는 이런 것이었어요**


- 밀리의 서재 앱의 책장 편집 기능은 사용 빈도가 높지만, **편집 기능 진입 경로, 책 선택 방식, 이동 후 피드백 구조**에서 반복적으로 사용성 문제가 발생하고 있었어요.
- 특히 UT와 리뷰 분석에서는
    - 기능 위치 인지 실패
    - Flick 제스처 오인
    - 이동 후 결과 확인을 위한 역행 행동
    
    같은 구조적 문제가 꾸준히 확인되었어요.
    
- 저희는 이 문제를 **예측 가능성·다중성·변화지시성** 기준으로 분석하고, 1차–개선–2차의 검증 루프를 통해 편집 기능 전체를 재설계했어요.

## **이 과정에서 이런 역량을 활용했어요**


- UI Guideline 분석(예측 가능성·다중성·변화지시성 기준)
- UT/아이트래킹 설계 및 데이터 해석
- 문제 구조화 및 병목(bottleneck) 정의
- 인터랙션 설계 및 프로토타이핑(페이퍼 → 디지털)
- 개선안 효과 검증을 위한 실험 설계
- IA/화면설계서 작성

## **프로젝트에서 이런 역할을 맡았어요**


- 전체 사용성 평가와 분석 흐름을 설계했어요.
- 1차·2차 UT, 아이트래킹 지표를 직접 설계하고 데이터를 수집했어요.
- Flick 제스처 오인, 잘못된 진입 경로, 이동 피드백 부족이라는 핵심 문제를 구조적으로 정의했어요.
- 전체 선택, 책장 병합, 이동 Action Sheet 같은 핵심 기능의 UX 흐름을 설계했어요.
- 페이퍼와 디지털 프로토타입과 문서를 작성했어요.

## **프로젝트를 진행하며 이런 문제가 보였어요**


### **예측 가능성 문제**

- 편집 기능이 더보기 메뉴 안에 있어 기능 진입 시간이 지연되었어요.
- 참가자 5명 중 4명이 **잘못된 경로(책 옆 더보기 버튼)**로 편집 기능을 찾았어요.

### **다중성 문제**

- 책을 한 권씩 선택해야 했고, 60%가 **Flick 다중 선택이 될 것이라고 기대**했어요.
- 실제로는 지원되지 않아 반복 조작이 발생했어요.

### **변화지시성 문제**

- 책 이동 후 어디로 이동했는지 명확하지 않았어요.
- 일부 사용자는 다시 되돌아가 변경 사항을 직접 확인해야 했어요.

## **그래서 이런 방식으로 문제를 바라봤어요**


- 문제를 단순한 UI 불편이 아니라 **사용자가 예상하는 패턴과 실제 기능 제공 방식의 불일치**로 정의했어요.
- 이를 확인하기 위해 다음 질문을 중심으로 분석했어요.
    - 사용자는 어떤 경로를 ‘편집 기능’이라고 예상할까?
    - 왜 Flick 제스처를 시도할까?
    - 시선은 어떤 영역에 집중되고, 어떤 영역을 보지 않을까?
    - 어떤 지점이 Task 시간 증가의 핵심 병목일까?
- 이 질문들을 UT와 아이트래킹을 병행하여 데이터 기반으로 검증했고, 그 결과 기능 진입 위치, 다중 선택 부재, 피드백 위치가 핵심 병목이라는 것을 확인했어요.

## **이 문제를 이렇게 해결해봤어요**


### **편집 기능의 예측 가능성 개선**

- 더보기 안에 있던 편집 기능을 네비게이션 바로 이동했어요.
    
    → 기능 직접 진입률 **40% → 80% 증가**
    

### **다중 선택 기능 제공**

- 전체 선택 기능을 도입해 단일 선택 조작을 제거했어요.
    
    → Flick 오인율 **60% → 0%**
    
    → 평균 Task 시간 **36.7초 → 15.6초(57% 감소)**
    

### **이동 과정의 변화지시성 강화**

- “n권을 ‘책장명’ 책장으로 옮겼어요 | 되돌리기” 형태의 Toast 메시지를 추가했어요.
- 사용자는 이동 결과 확인을 위해 되돌아갈 필요가 없어졌어요.

### **책장 병합 기능 제공**

- 다중 책장 병합 기능과 새 책장 생성 플로우를 설계했어요.
    
    → 여러 책장을 한 번에 정리할 수 있게 되었어요.
    

### **Gmail Snackbar 패턴의 맹점 발견**

- Gmail의 Snackbar UI를 참고해 하단 Toast를 설계했지만, 아이트래킹 결과 사용자의 시선이 하단 영역에 거의 머물지 않았어요.
- 이를 통해 **레퍼런스 UI가 항상 동일한 효과를 보장하는 것은 아니라는 점**을 확인했어요.
- 피드백의 위치는 사용자의 시선 흐름을 기준으로 재검토해야 한다는 점을 배웠어요.

## **이 경험을 통해 이런 것들을 배웠어요**


### **사용성 테스트는 ‘우리가 보기 좋은 문제’가 아니라 ‘사용자의 문제’를 찾는 과정이라는 점을 배웠어요.**

- 초기에는 무엇이 테스트할 가치가 있는지 판단하기 어려웠지만, 사용자의 행동을 관찰하면서 **겉으로 보이지 않는 문제를 발견하는 것이 핵심**이라는 점을 알게 되었어요.

### **사용자는 우리가 예상하지 못한 방식으로 제품을 다룬다는 사실을 다시 이해했어요.**

- 일부 참가자는 전혀 예상하지 못한 경로와 방식으로 앱을 사용했어요.
- 이를 통해 **기획자의 가정보다 실제 행동을 보는 것이 더 중요**하다는 점을 배웠어요.

### **아이트래킹은 단순 행동만으로는 보이지 않는 문제를 드러낸다는 점을 배웠어요.**

- Gmail 앱의 Snackbar 패턴을 그대로 적용했지만, 사용자의 시선이 하단 Toast에 머물지 않는다는 결과**를 보았어요.**
- 유명한 레퍼런스 패턴도 맥락이 다르면 효과가 없을 수 있다는 사실을 체감했어요.

### **테스트 환경을 설계하는 것도 기획자의 책임이라는 점을 배웠어요.**

- 마스크, 머리카락, 조명 등으로 일부 데이터가 제대로 수집되지 않았어요.
- 이 경험을 통해 **테스트 환경·장비 변수도 결과의 일부**라는 것을 확실히 배웠어요.

### **기업은 ‘문제가 있어 보이는 기능’이 아니라 ‘지표의 원인’을 찾기 위해 테스트를 수행한다는 관점을 얻게 되었어요.**

- 스티브 크룩의 저서에서 본 사례를 보며 사용성 테스트가 **지표 문제를 진단하는 도구**라는 점을 깨닫게 되었어요.
- 이를 통해 기업의 테스트 주제 선정 기준을 더 넓게 생각할 수 있었어요.
